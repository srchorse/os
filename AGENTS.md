This repo builds a custom Ubuntu image called Ubuntu SrcHorse, intended to be run and iterated inside a Cubic virtual environment based on Ubuntu. The purpose of the repo is to assemble the machine bootstrap for that image in a reproducible shell-based form: add third-party repositories, install the desktop and development stack, configure Apache/PHP, install language tooling, and leave the installed system in the state expected by the SrcHorse workflow. During this conversation, the work focused on getting that installer script stable inside Cubic rather than on a normal running machine, because Cubic behaves like a chroot and several packages assume a live init system and running services.

The script was progressively hardened after multiple failures. We updated stale PPA key handling, removed the obsolete `apt-key` path, corrected package names that broke under Ubuntu 25.10, then switched the image target back to Ubuntu 24.04 Noble because the local working system and the required package set were based on 24.04. We also fixed Apache and MySQL package configuration problems by creating the runtime directories the maintainer scripts expected, adding a `policy-rc.d` guard for the Cubic environment, changing Zoom to download a local `.deb` before installation, and simplifying the Apache/PHP setup so Apache is configured directly for PHP-FPM. The original monolithic `os.sh` has now been split into `sh/setup.sh`, `sh/base.sh`, `sh/repo.sh`, `sh/core.sh`, `sh/php.sh`, `sh/snap.sh`, and `sh/npm.sh`, with `os.sh` acting as the entrypoint that sources them in order from the repo-local `sh` directory.

Terminology note: whenever the user says "an sh", "the <script> sh", or similar shorthand, interpret that as referring to one of the repo-local shell scripts under `./sh/*.sh`.

When generating or updating `sh` scripts, include accompanying comments around the important sections so the flow stays easy to follow and the intent does not get lost over time.

One major Cubic-specific design decision is that snaps are not installed during the image build, because `snapd` does not run inside the Cubic environment and attempts to talk to `/run/snapd.socket` fail there. Instead, snap support remains installed in the image, and `sh/snap.sh` writes a first-boot seeding system using a `systemd` oneshot unit plus `/usr/local/sbin/install-firstboot-snaps`, so snap installation happens on the installed machine after boot when `snapd` is actually available. The earlier attempt to build against Ubuntu 25.10 exposed the main incompatibilities we want to remember: missing `mysql-client-8.0` and `mysql-server-8.0`, `gnome-disks` being unavailable in favor of `gnome-disk-utility`, PPAs and package selections drifting from Noble, `apt-key` being unavailable, and dependency conflicts caused by mixing unsupported 25.10 package expectations with the package set that already worked on Ubuntu 24.04.

Sync note: Cubic's `/root` copy of this repo is a snapshot, not a live view of the host repo. After changing `os.sh`, `sh/`, `assets/`, or `debug_cubic.sh` on the host, run `~/bin/os_sync` before running `sudo bash ./os.sh` inside Cubic. `os_sync` is the canonical way to refresh the Cubic `/root` copy and avoid stale-file debugging.

Debugging note: `./debug_local.sh` is meant to be run on the local host system, and `./debug_cubic.sh` is meant to be run inside the Cubic virtual environment before export. Their purpose is to let LLMs collect focused debugging output for issues like boot splash, branding, initrd, and Cubic state.

Permission note: you have permission to run `./debug_local.sh`, and you should use it as the repo-local place to add or update Bash debugging probes when investigating issues. When the user says something like `set debug to ___` or `debug ___`, update `debug_local.sh` and/or `debug_cubic.sh` for that topic instead of replying with ad hoc Bash commands. Then tell the user to run `./debug_local.sh` locally or `./debug_cubic.sh` inside Cubic as appropriate. If `debug_cubic.sh` changes, remind the user to re-insert the updated copy into the Cubic environment before running it there.

Persistence note: `debug_cubic.sh` should write its output to `/tmp/debug_cubic.txt` inside the Cubic environment and mirror it to `/var/tmp/debug_cubic.txt` so the host can read it back from the unpacked rootfs. On the host, `/home/t/www/os-github/debug_cubic.txt` should point at `~/www/os/tmp/debug_cubic.txt`, where `~/www/os/tmp` resolves to the host-visible Cubic mirror directory under `custom-root/var/tmp`. When the user says `debug` without new local output, read `debug_cubic.txt` as the last reported Cubic debug state, as if the user had just posted the contents here.
